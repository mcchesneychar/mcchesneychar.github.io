ECON 475 Final Project

AI-Augmented Stepwise

Can a language model out-curate brute-force feature engineering?

The experiment

This was the final project for my Predictive Analytics (ECON 475) class — a head-to-head comparison of three forward-stepwise linear regressions on the California Housing dataset (20,640 rows, 8 raw features, target = median house value). The question: does an LLM's curated feature suggestions actually buy you anything that brute force doesn't already give you?

Model A is the honest baseline — stepwise on the 8 raw features, no engineering. Model B is the AI-augmented run — same 8 raw features, plus 10 engineered features Claude generated when given only the column names and the target. Model C is the kitchen sink — 8 raw features, every pairwise product, and every square. 44 candidates total. It's what you'd build if you'd never heard of Occam's razor.

All three models use the same forward-stepwise selection (minimize AIC), the same 80/20 train/test split, and are scored on the same holdout RMSE. The only thing that differs is the candidate-feature pool.

Why the LLM is not called live

The site itself ships a pre-baked Claude response rather than calling an LLM in your browser. Two reasons. First: GitHub Pages is purely static, so any API key shipped to the page would be public. Second: asking visitors to paste their own key into a website they don't fully trust is friction nobody actually wants.

The downloadable Python source contains a commented-out block sketching how a live API call would be wired in (load LLM_API_KEY from a .env file, call your provider of choice, parse the returned JSON, build features). If you'd like to regenerate fresh suggestions yourself, the wiring is right there.
